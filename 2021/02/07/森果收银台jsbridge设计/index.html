森果收银台 jsBridge 设计

发表于 2021-02-07

前言

森果收银台中涉及到不少跨平台通信，如电子秤，打印机、副屏、钱箱等客户端才能调用的硬件，同时涉及到 android 和 windows 两个平台，如果不设计通信交互，那么代码将会变得十分混乱，因此一个好用的 jsBridge 设计对收银台来说尤为重要。

jsbridge 简介

jsBridge 指通过客户端以及 web 端定义的接口实现互相通信的桥梁，使用 jsBridge 客户端可以调用 web 端方法，反之亦然。因不同平台与 web 端通信方式不同，所以 jsBridge 还有一个作用在于抹平不同平台的差异，实现通信交互的一致性。

jsbridge 需注意的点

参数传递

首先是参数传递，参数一般只可传递简单的基础类型，最好是通过 json 来传递，利于跨平台以及参数扩展。

回调方法

同时客户端完成操作后可能需要传递参数给 web 端，所以需要定义回调方法，但需要根据场景来进行定义，否则可能会造成内存泄漏。

web 端 jsBridge 设计

android 与 js 通信

在讲解 web 端 jsbridge 之前，先了解一下 android 端如何与 js 通信，因为这涉及到 web 端 jsBridge 设计。
js 调用 android 方法，首先需要在 webview 中定义 javascriptInterface, 如下：

通过上述方法，即可在 web 端调用 android 函数

window.clientBridge.sendEventToClient(event, JSON.stringify(params || {}))

其中 clientBridge 是 webview 定义在 Window 上的全局对象，通过此对象可调用 android 方法。

android 执行 js 函数可通过以下方式调用

webView.loadUrl(“javascript:window.invokeFunction()”);

通过上述安卓代码可以得知 js 调用 android 的每一个方法都需要定义在 javascriptInterface 中，当然也可以通过 prompt 以及链接跳转的形式与 android 通信，这里不过多赘述。

每一个函数定义在 javascriptInterface 中会导致一个问题，如果有很多方法，那么 javascriptInterface 会有些混乱，代码不好维护，同时 android 调用 web 端的代码也容易出现以下问题。

  1. 如果 window 对象上定义过多全局函数不利于维护，还可能造成内存泄漏。
  2. 如果 web 端 window 对象上没定义这一函数，那么可能会报错。
  3. window 对象上的函数上下文可能访问不到调用客户端函数的上下文，导致代码组织混乱，更不利于维护，这是很关键的一点。

所以 web 端 jsBridge 封装需解决以上三个问题。

如果 window 上定义过多全局函数不利于维护，还可能造成内存泄漏

但如果不定义一个全局函数，那么 android 端如何调用到我们执行客户端方法时传递的回调函数呢？这样又回到了第一个问题。

如果 window 上定义过多全局函数不利于维护，还可能造成内存泄漏

对此我们可以这样解决。

  1. 将回调函数缓存起来
  2. 客户端告知 web 端你将要执行一个回调函数，并给出该回调函数相关的信息
